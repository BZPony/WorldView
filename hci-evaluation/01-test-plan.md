# WorldView 可用性测试计划

## 1. 测试背景

WorldView 是一个面向架空世界创作者的交互式地图与时间轴可视化原型。用户可以查看人物、地点、组织、政权等实体，并通过时间轴观察实体位置、轨迹和名称随时间变化的过程。

当前测试版本已经部署为线上静态网页：

```text
https://bzpony.github.io/WorldView/
```

测试时不要求被试安装、配置或理解代码，只测试浏览器中原型界面的可理解性和可操作性。

## 2. 测试目标

本轮测试关注以下问题：

1. 用户能否快速理解系统用途。
2. 用户能否理解地图、实体标记和时间轴之间的关系。
3. 用户能否通过地图或侧边栏找到目标实体。
4. 用户能否理解详情面板中的信息结构。
5. 用户能否发现并完成基本编辑操作。
6. 用户能否使用筛选功能缩小资源范围。
7. 用户能否完成创建地点、创建途径点、删除和撤销等操作。
8. 哪些交互最容易造成困惑、误操作或停顿。

## 3. 测试方法

推荐使用两阶段方法：

1. 先做 1 人 pilot test，检查链接、任务说明和问卷是否清楚。
2. 再做 3 到 5 人小范围正式试用。

正式测试优先采用“主持式可用性测试”：

- 主持人给出任务。
- 被试独立操作线上原型。
- 主持人观察并记录。
- 被试完成任务后填写问卷。
- 最后进行 3 到 5 分钟简短访谈。

如果无法现场观察，可以采用“远程自助试用 + 问卷”：

- 被试自行打开链接。
- 被试按简化任务试用。
- 被试填写问卷和开放题。
- 这种方式更适合快速收集主观反馈，但无法准确记录用时、错误和卡点。

## 4. 被试选择

本轮按课程实践的小范围试用处理，建议 3 到 5 人。被试可以从熟人、同学或朋友中选择，不需要公开招募。

| 类型 | 人数 | 选择理由 |
| --- | ---: | --- |
| 熟悉网页操作的普通用户 | 2 到 3 | 检查界面是否能被一般用户理解 |
| 有创作兴趣的用户 | 1 到 2 | 更接近架空世界创作者、写作者、跑团或游戏设定爱好者 |

测试前记录基本信息：

- 编号
- 年级或专业
- 是否有世界观创作经验
- 是否常用地图类应用
- 是否常用时间轴、项目管理或知识管理工具

样本较小时，报告中不强调统计显著性，而是结合任务表现、问卷平均分和用户原话分析主要问题。

## 5. 测试环境

- 页面：`https://bzpony.github.io/WorldView/`
- 浏览器：Chrome、Edge 或 Safari 均可。
- 设备：建议电脑或平板横屏，不建议第一轮使用手机。
- 网络：需要能访问 GitHub Pages、Leaflet CDN 和地图瓦片。

测试前检查：

- 页面能打开。
- 能看到地图、左侧侧边栏和底部时间轴。
- 地图瓦片能加载。
- 刷新页面后可恢复初始状态。

## 6. 测试流程

| 阶段 | 时间 | 内容 |
| --- | ---: | --- |
| 开场说明 | 2 分钟 | 告知测试目的、说明测试的是系统不是用户 |
| 背景询问 | 2 分钟 | 了解被试经验 |
| 任务测试 | 12 到 18 分钟 | 完成核心任务并记录表现 |
| 后测问卷 | 3 到 5 分钟 | 填写量表和开放题 |
| 简短访谈 | 3 到 5 分钟 | 追问卡点和改进建议 |

单人总时长控制在 20 到 30 分钟。

## 7. 成功指标

### 任务指标

- 任务完成率
- 平均完成时间
- 错误次数
- 是否需要主持人提示
- 停顿和犹豫位置

### 主观指标

- 整体易用性评分
- 时间轴理解评分
- 筛选功能有用性评分
- 编辑和创建流程清晰度评分
- 用户对继续使用意愿的评分

## 8. 提示规则

测试中主持人尽量不主动教用户怎么做。只有在以下情况给提示：

- 被试超过 60 秒没有进展。
- 被试明确表示无法继续。
- 被试操作可能导致任务无法继续。

提示分级：

| 等级 | 描述 | 示例 |
| --- | --- | --- |
| 0 | 无提示 | 用户独立完成 |
| 1 | 轻提示 | “你可以再看看侧边栏里有没有相关入口” |
| 2 | 中提示 | “这个任务可能需要在地图上右键” |
| 3 | 强提示 | 直接告诉用户具体按钮或操作 |

远程自助测试没有主持人提示，报告中可只统计问卷和开放题，不统计提示等级。

## 9. 伦理与说明

测试前应告知：

- 测试的是系统，不评价个人能力。
- 用户可以随时停止测试。
- 记录内容只用于课程项目分析。
- 如果引用用户原话，报告中使用匿名编号。
- 原型不会保存用户长期数据，刷新页面后会回到初始状态。

## 10. 预期风险

| 风险 | 影响 | 应对方式 |
| --- | --- | --- |
| 被试以为要看代码 | 干扰测试 | 只发线上链接，不发 GitHub 仓库链接 |
| 地图资源加载失败 | 无法完成地图任务 | 提前检查网络，准备截图或备用浏览器 |
| 右键创建不易发现 | 多人任务失败 | 记录为关键发现，不提前说明 |
| 单击编辑不易发现 | 编辑任务失败 | 观察用户是否尝试双击、找编辑按钮 |
| 数据刷新丢失 | 测试状态不一致 | 明确说明这是临时原型，刷新后恢复初始状态 |
| 手机端操作不顺 | 影响右键和时间轴任务 | 第一轮要求电脑浏览器参与 |
